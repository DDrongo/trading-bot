
# ДОКУМЕНТ 3: ТЗ ДЛЯ ФАЗЫ 1.0 — GIT

## 📌 КОНТЕКСТ (откуда взялось, почему это важно)

### Ситуация сейчас
Весь код проекта лежит у тебя локально. Это создаёт несколько проблем:
1. **Нет истории** — если что-то сломается, нельзя откатиться к рабочей версии
2. **Нет веток** — нельзя параллельно разрабатывать разные фичи
3. **Нет бэкапа** — если диск умрёт, код потеряется
4. **Нет автоматического деплоя** — на сервер код придётся переносить вручную

### Почему Git — первая задача
Git — это **фундамент всего остального**. Без него:
- Нельзя нормально тестировать (каждый тест должен быть привязан к коммиту)
- Нельзя откатываться при ошибках
- Нельзя автоматически обновлять код на сервере

### Что даст выполнение этой задачи
- ✅ Полная история всех изменений
- ✅ Возможность работать в отдельных ветках
- ✅ База для автоматического деплоя
- ✅ Защита от потери кода

### Шаг 2: Первый коммит
- [ ] Склонировать пустой репозиторий:  
  ```bash
  git clone https://github.com/твой-логин/trading-bot.git
  cd trading-bot
  ```
- [ ] Скопировать все файлы проекта в папку репозитория

- [ ] Создать файл `.gitignore` со следующим содержимым:
  ```
  __pycache__/
  *.pyc
  .env
  data/*.db
  logs/*.log
  config/config.yaml
  .vscode/
  .idea/
  *.db
  *.log
  ```

**Что здесь и зачем:**
- `__pycache__/` и `*.pyc` — скомпилированные файлы Python, не нужны в репозитории
- `.env` — файл с секретами (ключи API), должен быть только локально
- `data/*.db` — базы данных (не нужно хранить в Git, только бэкапы)
- `logs/*.log` — логи (слишком большие и часто меняются)
- `config/config.yaml` — конфиг с ключами (должен быть локально)
- Папки редакторов (`.vscode/`, `.idea/`) — личные настройки

- [ ] Создать файл `README.md` с кратким описанием:
  ```markdown
  # Trading Bot
  
  Торговый бот для Bybit с трёхэкранной стратегией Александра Элдера.
  
  ## Возможности
  - Анализ тренда (D1)
  - Поиск зон входа по Фибоначчи (H4)
  - Генерация сигналов по паттернам (M15)
  - Сохранение сигналов в БД
  - Мониторинг открытых позиций
  - Два режима: PAPER (виртуальный счёт) и LIVE (реальная торговля)
  
  ## Структура проекта
  - `main.py` — точка входа
  - `core/` — основные модули
  - `monitor_three_screen.py` — монитор сигналов
  - `config/` — конфигурация (не в Git)
  ```

- [ ] Выполнить первый коммит:
  ```bash
  git add .
  git commit -m "init: первая версия торгового бота"
  git push origin main
  ```

### Шаг 3: Структура веток

**Контекст:**  
Ветки нужны, чтобы разделять стабильный код и разработку.  
- `main` — только проверенный, стабильный код  
- `dev` — для разработки (можно экспериментировать)  
- `feature/*` — для конкретных задач (например, `feature/phase-1-package-1`)

- [ ] Создать ветку `dev` для разработки:
  ```bash
  git checkout -b dev
  git push origin dev
  ```

- [ ] Создать ветку для первого пакета тестирования:
  ```bash
  git checkout -b feature/phase-1-package-1
  git push origin feature/phase-1-package-1
  ```

### Правила работы с Git на проекте
- **Никаких прямых правок на сервере.** Любое изменение — через коммит в `dev`, потом тестирование, потом merge в `main`
- **Каждая фаза** — в отдельной ветке `feature/*`
- **Коммиты** — маленькие и с понятными описаниями (не "fix", а "исправлен расчёт SL в screen3")
- **Перед началом новой фазы** — создать свежую ветку от `dev`

### Если что-то пошло не так
Git — штука прощающая ошибки. Если случайно закоммитил лишнее или не туда — не страшно, всё можно откатить. Главное — не паниковать и спрашивать.
