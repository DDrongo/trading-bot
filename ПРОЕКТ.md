## 🎯 РАЗДЕЛ 4: ЦЕЛЕВОЕ СОСТОЯНИЕ (КУДА МЫ ИДЁМ)

### 4.1 Два режима работы

| Аспект | PAPER | LIVE | Почему |
|--------|-------|------|--------|
| Анализ рынка | ✅ | ✅ | Статистика должна быть сопоставима |
| Генерация сигналов | ✅ | ✅ | --//-- |
| Типы сигналов | ✅ | ✅ | --//-- |
| Исполнение ордеров | Симуляция | Реальное | Единственное различие |
| Учёт комиссий | ✅ (симуляция) | ✅ (реальные) | Чтобы PAPER не был "слишком идеальным" |
| Учёт проскальзываний | ✅ (симуляция) | ✅ (реальные) | --//-- |

### 4.2 Три типа сигналов

| Тип | Таймфрейм | R/R | Время жизни | Назначение | Статус |
|-----|-----------|-----|-------------|------------|--------|
| **WATCH** | H4 | — | — | Пре-сигнал (4/5 условий) | ❌ Фаза 1.2 |
| **LIMIT** | H4 | ≥ 3:1 | 24 часа | Среднесрочные входы в зону | ✅ Есть |
| **INSTANT** | M15 | ≥ 2:1 | 3 часа | Быстрые сделки внутри дня | ❌ Фаза 1.2 |

### 4.3 Полный цикл сделки
```
1. Генерация сигнала → событие TRADING_SIGNAL_GENERATED ✅ (Фаза 1.1)
2. Position Manager получает событие → открывает позицию ⏳ (Фаза 1.3)
3. Мониторинг цены → проверка TP/SL ⏳ (Фаза 1.3)
4. Закрытие → событие TRADE_CLOSED ⏳ (Фаза 1.3)
5. Расчёт PnL → сохранение в историю ⏳ (Фаза 1.4)
```

## 📌 РАЗДЕЛ 6: КЛЮЧЕВЫЕ РЕШЕНИЯ (КОНТЕКСТ)

| Решение | Почему |
|---------|--------|
| **EventBus интегрирован** | События публикуются из репозитория, оркестратор уведомлений не дублирует |
| **Три типа сигналов** | WATCH — вовлечение, LIMIT — основа, INSTANT — частота |
| **LIMIT R/R ≥ 3:1, INSTANT R/R ≥ 2:1** | Баланс между качеством и количеством |
| **PAPER и LIVE с одинаковой логикой** | Чтобы статистика была достоверной |
| **Пуш после каждой фазы** | Чёткая фиксация стабильных состояний |
| **Менеджер контролирует Git** | Единая точка ответственности за версионирование |
